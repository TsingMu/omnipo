# system-monitor 增量规范

## ADDED Requirements

### Requirement: 系统监控页必须提供五维度只读概览

系统 MUST 在主窗口“系统监控”页面提供 CPU、内存、能耗、磁盘和网络五个维度的整机级只读概览。每张卡片在数据可用时展示真实当前值，不可用时展示一致、可理解的降级原因；不得伪造数值、百分比或速率。

#### Scenario: 五维度全部可用

- **当** 用户进入“系统监控”页面
- **并且** 五个维度的采样均成功
- **那么** 页面同时显示 CPU、内存、能耗、磁盘、网络五张卡片
- **并且** 每张卡片展示对应维度的真实当前值
- **并且** 不展示虚构数字或演示百分比

#### Scenario: 单维度不可用

- **当** 五维度中某一维度采样失败
- **那么** 对应卡片显示降级原因
- **并且** 其他维度的真实值继续展示
- **并且** 不以 0 或空状态伪装成功

#### Scenario: 能耗维度边界

- **当** 系统监控页展示能耗卡片
- **那么** 卡片可显示电池电量与充放电状态（若设备有电池）
- **并且** 整机能耗瓦数不展示虚构数字
- **并且** 明确说明整机能耗需系统特权或私有 API，本应用不调用

### Requirement: 系统监控必须使用公开 API 且不申请额外权限

系统 MUST 在 App Sandbox 内完成全部采样，不得申请辅助功能、输入监控、完全磁盘访问或其他与系统资源监控无关的权限。CPU 使用 Mach `host_processor_info`；内存使用 Mach `host_statistics64` 与 `sysctl`；电池使用 `IOPSCopyPowerSourcesInfo`；网络使用 `getifaddrs`；磁盘容量复用 `DiskUsageService`。不得调用 SMC、PowerMetrics、`iostat` 子进程或其他私有/特权 API。

#### Scenario: 首次使用系统监控

- **当** 用户首次进入“系统监控”页面
- **那么** 系统不弹出任何隐私权限请求
- **并且** 不读取其他应用的状态或文件内容
- **并且** 不调用任何子进程

#### Scenario: 维度因 API 不可用降级

- **当** 某公开 API 在当前系统不可用或被 Sandbox 拒绝
- **那么** 对应卡片显示降级原因
- **并且** 不绕过 Sandbox、不申请额外权限

### Requirement: 系统监控必须支持可取消的受控采样

系统 MUST 仅在“系统监控”页面活跃时进行后台采样；用户离开页面或关闭主窗口时立即停止采样。采样必须支持取消，旧采样任务不得在新任务启动后继续推送结果。

#### Scenario: 进入页面启动采样

- **当** 用户进入“系统监控”页面
- **那么** 系统启动后台采样任务
- **并且** 立即推送一次首帧
- **并且** 后续按配置间隔定期采样

#### Scenario: 离开页面停止采样

- **当** 用户切换到其他页面或关闭主窗口
- **那么** 系统立即取消后台采样任务
- **并且** 不再推送任何快照
- **并且** Omnipo 自身 CPU 占用回落，不构成后台监控负担

#### Scenario: 快速刷新与频率切换

- **当** 用户连续点击刷新或快速切换采样间隔
- **那么** 旧采样任务被取消
- **并且** 只接受当前采样代次的结果
- **并且** 旧结果不会覆盖新结果

### Requirement: 采样间隔必须可配置且范围受限

系统 MUST 允许用户在“系统监控”页面或设置中配置采样间隔，默认值为 5 秒，可调范围为 1 至 30 秒。系统不得接受 0、负值或超过上限的间隔。

#### Scenario: 调整采样间隔

- **当** 用户将采样间隔改为合法范围内的值
- **那么** 新间隔在下次采样周期生效
- **并且** 设置在重启后保留

#### Scenario: 输入非法间隔

- **当** 用户尝试设置 0、负值或超过 30 秒的间隔
- **那么** 系统拒绝该值
- **并且** 保持上一次有效间隔
- **并且** 不破坏采样稳定性

#### Scenario: 损坏配置回退

- **当** 应用启动时读取到损坏或越界的采样间隔设置
- **那么** 系统不崩溃
- **并且** 回退到默认 5 秒
- **并且** 日志只记录稳定错误代码

### Requirement: 系统监控必须保护用户隐私

系统 MUST 保证所有采样的指标仅在内存中保留短暂快照，不写入磁盘、不上传网络、不进入 OSLog 的数值细节。日志只记录稳定事件码（启动、停止、采样失败）与维度缩写，不得记录具体数值、网络接口名、电池百分比、内存字节数或 CPU 百分比。

#### Scenario: 普通采样

- **当** 系统按间隔采样五个维度
- **那么** 指标仅用于当前 UI 展示
- **并且** 不写入磁盘或数据库
- **并且** 不向任何网络服务发送数据

#### Scenario: 记录采样事件

- **当** 系统记录采样启动、停止或失败事件
- **那么** 日志只包含稳定事件码与维度缩写
- **并且** 不包含具体数值、接口名、电池百分比等可识别信息

#### Scenario: 关闭页面

- **当** 用户离开“系统监控”页面
- **那么** 系统停止采样
- **并且** 不保留历史趋势或快照缓存（除非后续 change 显式启用）

### Requirement: 系统监控磁盘卡必须复用 Disk Usage 数据源

磁盘卡 MUST 与 Dashboard、Disk Analysis 共享同一 `DiskCapacityAvailability` 状态来源，避免重复扫描与状态分裂。本 change 不实现实时磁盘 IOPS；磁盘卡仅展示容量摘要，IOPS 不可用时不伪造数据。

#### Scenario: 磁盘卡数据一致

- **当** 用户在 Dashboard、Disk Analysis 与 System Monitor 之间切换
- **那么** 三处的启动卷容量数字一致
- **并且** 不重复触发卷元数据读取

#### Scenario: IOPS 不可用降级

- **当** 磁盘卡展示读写吞吐
- **那么** 卡片明确说明实时 IOPS 需私有 API，本 change 不实现
- **并且** 不伪造吞吐数字

### Requirement: 系统监控必须支持无障碍与外观适配

系统监控页 MUST 使用系统语义色与材质，自动适配浅色与深色外观，并为每张卡片提供 VoiceOver 标签与键盘可达性。状态差异不得仅依赖颜色。

#### Scenario: 切换浅色与深色外观

- **当** 系统外观发生变化
- **那么** 五张卡片与降级文案保持可读
- **并且** 选中态、占用态、警告态不依赖颜色差异

#### Scenario: 使用 VoiceOver 浏览

- **当** 视障用户使用 VoiceOver 浏览系统监控页
- **那么** 每张卡片有清晰的语音标签
- **并且** 占用百分比、电池状态、网络速率均可被读出
- **并且** 降级原因可被读出
