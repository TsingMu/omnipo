# Change：添加 Permission Audit 核心能力

## 为什么需要此变更

Omnipo 已完成应用基础、Launcher、Dashboard、磁盘分析、System Monitor 和 Clipboard，但“权限审计”仍停留在占位页面。用户现在只能看到一个承诺“只读枚举已授权能力”的入口，却无法实际了解本机上哪些应用已经被授予了相机、麦克风、通讯录、日历、提醒事项、辅助功能或完全磁盘访问等敏感权限。

这项能力与 Omnipo 当前“本地优先、透明可控、安全保守”的方向高度一致：它不删除数据、不修改系统状态，核心价值在于把已经存在的权限暴露给用户看清楚。但 macOS 的 TCC 与权限体系没有完整稳定的公开高层 API，可读取性、字段结构和不同权限类别的查询方式都存在版本差异。因此本 change 必须先把“只读、版本容错、不可读取显式展示、绝不误报为未授权”的原则正式落到规范里，再进入实现。

## 变更内容

- 新增 `permission-audit` 能力规范、模型、服务协议与只读查询边界。
- 在 Omnipo 主窗口中提供真实的 Permission Audit 页面，替换现有占位页。
- 以只读方式枚举“应用 × 权限类别 × 当前状态”的本地审计结果，不读取隐私内容本身。
- 为不同权限类别建立统一展示模型，例如：
  - 相机
  - 麦克风
  - 照片
  - 通讯录
  - 日历
  - 提醒事项
  - 辅助功能
  - 完全磁盘访问
- 支持按权限类别和应用过滤结果，并展示明确的状态说明。
- 当某类权限在当前 macOS 版本、沙盒约束或本机环境下不可读取时，显式展示“不可读取”及原因，不将其误报为“未授权”。
- 优先复用 Omnipo 已有基础设施：
  - `SettingsService`
  - `LoggingService`
  - 现有主窗口导航与 Launcher 命令路由
  - `Infrastructure/Permissions` 目录中的只读实现边界

## 影响范围

### 新增能力

- `permission-audit`：提供应用隐私权限的本地只读审计视图和不可读取说明。

### 修改的现有能力

- `application-foundation`：新增权限审计所需依赖装配和统一模型落点。
- `launcher`：继续复用“权限审计”命令入口，但命令目标从占位页升级为真实功能页。

## 非目标

- 不修改其他应用的 TCC 授权记录。
- 不写入或重置任何权限状态。
- 不读取相机画面、麦克风音频、通讯录条目、日历内容、提醒事项内容、照片内容或其他隐私数据本身。
- 不申请“为了审计而新增”的高权限授权。
- 不把读取失败、数据库不可访问、系统版本不支持或沙盒限制误报为“未授权”。
- 不承诺在所有权限类别上都能完整枚举所有第三方应用。
- 不实现权限变更操作、批量重置、修复建议自动执行或一键跳转修改。

## 用户价值

- 用户可以集中查看本机应用已经拿到的敏感权限，而不必逐项翻系统设置。
- 用户可以区分“已授权 / 未授权 / 未知 / 不可读取”，减少误判。
- 用户可以知道当前结果来自本地只读审计，不会触发新的隐私访问。
- Omnipo 的“权限审计”入口从占位页升级为真正可用的系统诊断工具。

## 风险概述

- TCC 相关数据结构可能随 macOS 版本变化，读取逻辑需要版本容错。
- 某些权限类别没有统一稳定的公开枚举 API，可能必须部分降级。
- App Sandbox、系统数据库访问权限和不同权限类别的实现方式差异，会导致“有些能读、有些不能读”的混合状态。
- 权限状态本身属于敏感元数据，日志与错误输出必须避免泄漏应用名、路径和底层细节。

详细风险与降级策略见 `design.md`。

## 成功标准

- Permission Audit 页面不再显示占位文案，而能展示真实审计结果或明确的不可读取状态。
- 审计结果仅反映权限授权状态，不读取任何隐私内容本身。
- 用户可以按权限类别和应用过滤结果。
- 无法读取的权限类别必须展示原因，不误报为“未授权”。
- 日志不包含应用路径、应用名列表、底层数据库内容或其他敏感上下文。
- 工程持续可编译，测试通过，人工验收通过后再合并规范并归档 change。
