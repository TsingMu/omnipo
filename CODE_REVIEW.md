### REVIEW-001 — P1：日志命令被同名 Shell 函数遮蔽

- 文件：`script/build_and_run.sh`
- 位置：第 71–81 行
- 状态：**已修复(2026-06-19)**

脚本定义了 `log()` 函数，因此 `stream_logs` 和 `run_telemetry` 中的 `log stream`、`log show` 实际调用的是这个 Shell 函数，只会打印传入参数，不会执行 macOS 统一日志命令。

影响：

- `logs` 模式不会流式读取日志。
- `telemetry` 模式不会查询历史日志。
- `tasks.md` 中"支持 logs、telemetry 模式"的完成状态与实际行为不一致。

建议：

- 将输出辅助函数改名为 `print_log`；或
- 在系统日志调用处显式使用 `/usr/bin/log stream` 和 `/usr/bin/log show`。
- 增加脚本级冒烟测试，验证系统命令确实被调用。

修复与验证：

- 辅助函数更名为 `note`,所有原 `log` 调用迁移至 `note`。
- 新增常量 `LOG_BIN="/usr/bin/log"`,`stream_logs` 与 `run_telemetry` 使用 `"$LOG_BIN" stream` / `"$LOG_BIN" show` 显式调用系统日志命令。
- `grep -nE '^\s*log\s|^\s*log$' script/build_and_run.sh` 无匹配,确认无残留同名函数调用。

### REVIEW-002 — P1：日志脱敏允许部分文件路径公开写入 OSLog

- 文件：`App/Services/LoggingService.swift`
- 位置：第 68–99 行
- 关联文件：`App/Infrastructure/Logging/OSLogLoggingService.swift` 第 13–28 行
- 状态：**已修复(2026-06-19)**

当前脱敏仅识别 `/Users/`、`/private/var/` 和 `file://`。以下内容可能原样通过：

- `/Volumes/Backup/secret.pdf`
- `/tmp/private.txt`
- 单独出现的敏感文件名
- 使用未知 key 传入的剪切板或微信数据

这些动态值随后以 OSLog `.public` 隐私级别输出，不符合项目"不记录用户路径、文件名和隐私内容"的规范。

建议：

- 日志上下文使用明确允许的字段和值类型，避免任意 `[String: String]`。
- 动态值默认使用 `.private`，稳定事件名和错误代码才使用 `.public`。
- 将字符串脱敏视为最后一道防线，不作为主要隐私边界。
- 添加 `/Volumes/`、`/tmp/`、`~/`、文件名和未知字段测试。

修复与验证：

- `PrivacyRedaction` 扩展:
  - `forbiddenPathSubstrings` 新增 `/Volumes/`、`/tmp/`、`/private/tmp/`、`/var/tmp/`、`/dev/`、`/.Trash/`、`~/`、`${HOME}`。
  - 新增 `allowedContextKeys` 白名单;未列入白名单的 key 一律脱敏为 `<redacted>`。
  - 新增 `looksLikePath(_:)` 启发式:覆盖绝对路径前缀、`forbiddenPathSubstrings`、文件扩展名(`.pdf`/`.docx`/`.zip`/`.jpg`/`.mp4`/`.sqlite` 等)。
  - `forbiddenKeys` 扩充 `secret`/`token`/`password`。
- `OSLogLoggingService` 改造:`code` 与稳定 message 以 `privacy: .public` 输出,动态 context 字符串以 `privacy: .private` 输出,即便字符串脱敏漏过 OSLog 也会以 `<private>` 落盘。
- `LoggingServiceTests` 新增 13 个用例,覆盖 `/Volumes/`、`/tmp/`、`~/`、文件名、未知 key、白名单键、禁止键遍历、消息脱敏。
- `xcodebuild test` 41 passed / 0 failed。

### REVIEW-003 — P1：TaskProgress 的状态约束可被直接绕过

- 文件：`App/Models/TaskProgress.swift`
- 位置：第 21–27 行
- 状态：**已修复(2026-06-19)**

`status`、`completedUnits`、`totalUnits` 和 `error` 都是 `public var`。调用方可以在初始化后直接制造非法状态，例如：

```swift
var progress = TaskProgress(taskKey: "scan")
progress.totalUnits = 10
progress.completedUnits = 100
progress.status = .failed
progress.error = nil
```

这会绕过初始化器的归一化和 `validate` 校验，破坏下列规范约束：

- 完成量不得超过总量。
- failed 状态必须携带错误。
- 终态应保持稳定。

建议：

- 优先改为不可变值模型；或
- 使用 `public private(set)` 并提供 `updatingProgress`、`completing`、`failing`、`cancelling` 等受控转换方法。
- 为终态转换和非法突变增加测试。

修复与验证：

- `status`、`stage`、`userMessage`、`completedUnits`、`totalUnits`、`error` 全部改为 `public private(set) var`,`id`/`taskKey`/`isCancellable` 保持 `let`。
- 新增受控转换方法:`markRunning(stage:userMessage:)`、`updateProgress(completed:total:userMessage:)`、`markCompleted(userMessage:)`、`markFailed(_:userMessage:)`、`markCancelled(userMessage:)`,均返回 `Bool` 表示转换是否被接受。
- 所有转换方法在 `isTerminal == true` 时拒绝修改,保证终态稳定;`updateProgress` 在新 completed 超过 total 时拒绝修改。
- `ModelTests` 新增 4 个用例:非法 `updateProgress` 被拒、`markFailed` 要求 error、终态稳定性、cancelled 终态。
- `xcodebuild test` 41 passed / 0 failed。
