# Design：add-launcher-workbench

## 设计目标

本 change 的核心不是再造一个 Launcher，而是把已经可用的 Launcher 能力安全地投射到主窗口“快速启动”页中。设计上优先复用已有状态源、结果列表、输入桥接与执行路径，避免浮动面板和主窗口页出现两套不同规则。

## 能力范围

本阶段覆盖：

- 主窗口“快速启动”页直接展示 Launcher 搜索框和结果列表。
- 复用现有 `LauncherStore` 作为唯一查询状态源。
- 复用现有 `LauncherSearchField`、结果行和应用图标缓存。
- 复用现有结果执行器执行命令、应用与文件结果。
- 页面内保留安全错误反馈与空状态说明。

本阶段不覆盖：

- 新的搜索提供者或新的结果类型。
- 主窗口页与浮动面板之间的独立状态隔离。
- 浮动面板样式重做或复杂动画。

## 结构方案

### 共享内容组件

现有 `LauncherPanelView` 已包含成熟的：

- 搜索输入区
- 空状态
- 结果列表
- 局部失败提示
- 瞬态错误提示

因此本 change 采用“共享内容组件 + 两种承载样式”的方式：

- 独立浮动面板继续使用面板样式
- 主窗口“快速启动”页使用内嵌工作台样式

建议把现有内容视图扩展为支持 `panel` 与 `embedded` 两种展示模式：

- `panel`
  - 保持现有 560×420 浮动面板布局
  - 保持圆角、边框、固定尺寸
- `embedded`
  - 去掉“这是占位页”的语义
  - 放宽宽高约束，作为主窗口 detail 区中的主内容卡片
  - 保留相同的搜索与结果交互

## 状态与执行

### 查询状态

- 主窗口工作台直接复用 `LauncherCoordinator.store`
- 仍由 `LauncherStore` 作为唯一事实来源
- 进入工作台页面后，可主动触发空查询，让六个内置命令立即可见

### 结果执行

当前浮动面板执行结果的路径为：

- `LauncherPanelView` -> `LauncherCoordinator` -> `LauncherResultExecutor`

主窗口工作台应复用同一执行链路，但成功后不隐藏主窗口页。建议在 `LauncherCoordinator` 中抽取“内联执行”入口：

- 成功：保留当前页面，由命令/应用/文件执行各自生效
- 失败：把错误回写到 `LauncherStore.transientError`

### 取消行为

- 独立面板中的 Escape 仍用于关闭面板
- 主窗口工作台中的 Escape 不关闭主窗口，可仅清空查询和瞬态错误，或保持无副作用

## UI 行为

主窗口“快速启动”页应包含：

- 标题与简短说明，明确这是本地 Launcher 工作台
- 内嵌搜索工作台
- 对全局快捷键 `Option + Space` 的轻量提示

页面不得再显示 `Phase 0 暂未实现` 卡片，因为该页面展示的并不是未实现能力，而是已可用能力的主窗口承载形态。

## 风险与取舍

### 共享同一 Store

由于主窗口工作台与浮动面板共用 `LauncherStore`，若两者同时可见，查询和选择会同步变化。这是当前阶段接受的取舍，因为：

- 规则保持一致，避免两套搜索状态分叉
- 实现边界最小，风险低
- 用户通常不会长期同时操作两个 Launcher 视图

若未来用户体验证明需要隔离，再通过后续 change 引入双 Store 或会话模型。

## 测试与验收方向

- 单元测试覆盖工作台三种基础状态：空查询、可见结果、错误反馈展示模型不回退到占位语义。
- 人工验收覆盖：
  - 进入“快速启动”页后不再看到 Phase 0 占位文案
  - 空查询可见六个功能命令
  - 搜索应用/文件可正常返回结果
  - 点击或回车执行结果与浮动面板行为一致
