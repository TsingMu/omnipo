# Proposal：add-disk-analysis

## 背景

当前 Omnipo 的 Dashboard 仍将启动磁盘状态卡固定显示为“尚未扫描”，这是在尚未定义真实磁盘能力前的保守占位方案。随着 Launcher 已经提供“扫描磁盘”入口，主窗口首页和磁盘清理页需要进入下一阶段：先提供只读、低风险的磁盘容量概览，让用户在应用启动时立即看到当前启动卷的已用空间、可用空间和总容量，再逐步扩展到大文件展示、目录分析与清理建议。

## 问题

- 首页无法反映当前机器的真实磁盘状态，用户打开应用后看不到最基础的容量信息。
- “磁盘清理”目的地仍停留在占位说明，缺少明确的能力边界和后续扩展路径。
- 现有 Dashboard 规范要求“不展示虚构容量”，但没有说明在何种只读边界内可以展示真实容量。
- 用户无法快速识别最占空间的大文件，缺少按体积排序的文件名、路径和大小信息来辅助判断后续处理方向。

## 目标

- 新增 `disk-analysis` capability 的 OpenSpec 文档，定义只读磁盘容量概览的首个可交付范围。
- 将 Dashboard 启动磁盘状态卡升级为真实容量摘要，在应用启动时显示已用空间、可用空间和总容量。
- 明确磁盘分析页与首页卡片共享同一只读数据来源，不触发全盘扫描、不读取文件内容、不申请额外权限。
- 在磁盘分析页新增“大文件”列表，按大小降序展示文件名、路径和大小等只读元信息。
- 为后续目录级分析、清理建议和安全删除留出明确演进边界。

## 非目标

- 不在本 change 中实现重复文件识别、缓存/日志自动分类或基于内容的文件分析。
- 不实现文件删除、移动到废纸篓或批量清理动作。
- 不读取用户文件内容，不建立自有全盘索引，不后台持续扫描整个磁盘。
- 不将“无法读取”伪装成“空间为空”或“已扫描完成”。

## 受影响能力

- `main-dashboard`：首页启动磁盘状态卡改为显示真实容量摘要与只读状态说明。
- `disk-analysis`：新增磁盘分析 capability 规范，定义启动卷容量概览、大文件展示、手动刷新、降级文案和安全边界。

## 成功标准

- Omnipo 主窗口首次显示 Dashboard 时，状态卡在可用情况下展示启动卷的已用空间、可用空间和总容量。
- Dashboard 状态卡在加载态、可用态、不可用态之间切换时，文案和数字口径保持一致：加载中不显示伪造数字，不可用时不显示演示占用率。
- 若容量信息暂不可得，首页与磁盘分析页都显示一致且可理解的降级原因。
- Dashboard 快捷入口继续只承担导航职责，不因容量摘要刷新而自动触发扫描、清理或跳转。
- 规格明确要求容量摘要只依赖卷级元数据，不读取文件内容、不触发目录扫描。
- 当大文件列表可用时，页面按文件大小降序展示文件名、路径和大小，并且不读取文件内容。
- 当大文件列表暂不可用时，页面显示清晰降级说明，而不是伪造文件结果。
- 后续实现任务按单一 change 可顺序推进，并可在保持工程可编译的前提下逐步验证。
