# application-foundation 规范

## 元信息

- **能力名称**:application-foundation
- **首次实现 change**:`initialize-project-foundation`(Phase 0)
- **验收日期**:2026-06-19
- **当前状态**:已实现并合并

## Purpose

application-foundation 提供 Omnipo 的应用入口、桌面导航、依赖装配、横切服务(Settings、Logging)和统一模型(AppError、TaskProgress、OperationLog)。它是后续 Launcher、Clipboard、Dashboard、Cleaner、Uninstaller、Permission Audit、WeChat Manager、System Monitor 等能力的共同基线。
## Requirements
### Requirement: 应用必须提供原生 macOS 主窗口

系统 MUST 以 SwiftUI Scene 和 `NavigationSplitView` 管理可调整尺寸的主窗口。窗口必须依据当前实际内容布局建立标题栏安全边界；导航选择变化不得改变无关区域的空间位置，也不得在启动阶段执行敏感业务操作。

#### Scenario: 首次启动应用

- **当** 用户启动 Omnipo
- **那么** 系统显示主窗口
- **并且** 默认选择 Dashboard，或按明确的本地恢复设置显示上次目的地
- **并且** 启动过程不请求额外隐私权限

#### Scenario: 调整窗口尺寸

- **当** 用户调整主窗口尺寸
- **那么** 侧边栏和详情区域保持可用
- **并且** 内容不得因固定的移动端布局而不可访问

#### Scenario: 切换主窗口目的地

- **当** 用户通过侧栏、Launcher 命令或恢复状态切换目的地
- **那么** 主窗口保留相同的侧栏与详情内容原点
- **并且** 选择变化不会造成导航整体跳动或详情顶部裁切
- **并且** 不触发其他功能的后台任务

### Requirement: 应用必须提供稳定的功能导航

系统 MUST 使用适合 macOS 的侧边栏与详情布局,并至少提供 Dashboard、Launcher、Clipboard、Cleaner、Uninstaller、Permission Audit、WeChat Manager 和 System Monitor 八个入口。

#### Scenario: 选择功能入口

- **当** 用户选择任一侧边栏入口
- **那么** 详情区域显示对应页面或明确的未实现占位内容
- **并且** 不触发其他功能的后台任务

#### Scenario: 切换浅色与深色外观

- **当** 系统外观发生变化
- **那么** 侧边栏、文字和详情内容使用系统语义样式保持可读

### Requirement: 应用设置必须位于独立设置窗口

系统 MUST 提供 macOS 标准 Settings Scene。设置窗口必须与主功能导航分离,并通过本地设置服务持久化轻量偏好。

#### Scenario: 打开设置

- **当** 用户通过系统应用菜单打开设置
- **那么** 系统显示独立设置窗口
- **并且** 主窗口当前导航选择保持不变

#### Scenario: 保存轻量设置

- **当** 用户修改受支持的通用设置
- **那么** 设置服务仅在本地持久化该值
- **并且** 不上传该设置或关联用户身份

### Requirement: UI 必须通过服务协议访问横切能力

UI MUST 依赖 `SettingsService` 和 `LoggingService` 协议,不得直接创建其基础设施实现。具体实现必须由应用组合根统一装配。

#### Scenario: 创建应用依赖

- **当** 应用建立根视图
- **那么** 组合根提供设置与日志服务
- **并且** 根视图不直接创建 `UserDefaults` 或 `OSLog.Logger` 实现

#### Scenario: 运行隔离测试

- **当** 测试为 UI 或用例提供替代服务实现
- **那么** 测试无需访问用户真实设置或系统日志即可运行

### Requirement: 设置服务必须提供本地、类型安全的轻量配置

`SettingsService` MUST 为受支持的设置提供明确键和默认值。服务不得被用作剪切板历史、扫描结果或其他大体积数据的通用数据库。

#### Scenario: 读取尚未保存的设置

- **当** 请求一个尚未被用户修改的设置
- **那么** 服务返回规范定义的默认值

#### Scenario: 测试设置隔离

- **当** 单元测试写入测试专用设置存储
- **那么** 用户标准设置域不发生变化

### Requirement: 日志服务必须保护用户隐私

`LoggingService` MUST 使用本地统一日志并支持结构化等级和分类。日志不得包含剪切板内容、用户文件路径、文件名、微信数据或隐私内容。动态值必须以 OSLog `.private` 隐私级别输出,字符串脱敏作为兜底。

#### Scenario: 记录普通应用事件

- **当** 应用记录启动、导航或服务状态事件
- **那么** 日志包含模块、等级和安全事件描述
- **并且** 不包含被禁止的隐私字段

#### Scenario: 记录底层失败

- **当** 系统 API 返回可能包含敏感路径的错误
- **那么** 日志记录稳定错误代码和脱敏上下文
- **并且** 不直接输出敏感底层描述

### Requirement: 系统必须提供统一错误模型

系统 MUST 使用 `AppError` 表达用户取消、权限不足、资源不可访问、系统 API 失败、数据异常和未知失败。错误必须包含稳定代码、用户可理解的描述和可选恢复建议。

#### Scenario: 权限不足

- **当** 未来服务因系统权限不足无法完成操作
- **那么** 错误明确表示权限不足
- **并且** 提供安全、可执行的恢复建议
- **并且** 不把该失败误报为无数据

#### Scenario: 用户取消任务

- **当** 用户取消可取消任务
- **那么** 系统使用取消错误或取消状态结束任务
- **并且** 不将取消展示为未知故障

### Requirement: 系统必须提供统一任务进度模型

系统 MUST 使用可发送的值模型表达任务标识、阶段、状态、进度、取消能力和可选错误。模型必须同时支持已知总量与未知总量任务。可变字段必须以受控转换方法暴露,不允许调用方直接制造非法状态。

#### Scenario: 报告确定进度

- **当** 服务知道已完成工作量与总工作量
- **那么** 进度模型提供有效的完成比例
- **并且** 完成量不得超过总量

#### Scenario: 报告未知总量进度

- **当** 服务无法预先确定扫描总量
- **那么** 进度模型表达不确定进度
- **并且** UI 不显示虚假的百分比

#### Scenario: 任务完成或失败

- **当** 任务进入 completed、failed 或 cancelled 状态
- **那么** 该状态被视为终态
- **并且** failed 状态携带可展示的 `AppError`
- **并且** 终态后任何状态变更都被拒绝

### Requirement: Phase 0 必须保持最小权限面

基础应用 MUST NOT 读取 TCC 数据库、微信数据、剪切板历史或用户文件,不得执行扫描、清理、卸载或高频监控,也不得申请这些能力所需的额外隐私权限。

#### Scenario: 检查基础版本行为

- **当** 用户运行仅完成 Phase 0 的应用
- **那么** 应用只展示基础 UI 和本地通用设置
- **并且** 不读取或修改用户敏感数据
- **并且** 不执行任何删除行为

### Requirement: 系统集成设置必须反映有效状态

应用 MUST 将由 macOS 管理的系统集成状态与普通本地偏好区分开。设置界面必须显示系统当前有效状态；只有系统操作成功后才可以显示并持久化请求状态。失败、需要批准或不支持时必须提供安全、可执行的说明。

#### Scenario: 读取开机启动状态

- **当** 用户打开设置窗口
- **那么** 开机启动开关反映 `SMAppService` 的有效状态
- **并且** 读取状态不会自动注册或注销应用
- **并且** 本地缓存值不得覆盖系统有效状态

#### Scenario: 更新开机启动失败

- **当** 用户请求启用或关闭开机启动
- **并且** macOS 拒绝或无法完成请求
- **那么** 开关恢复到请求前的有效状态
- **并且** 应用显示安全的恢复建议
- **并且** 不持久化未生效的请求状态
- **并且** 日志不包含底层原始错误描述或用户路径

#### Scenario: 开机启动需要用户批准

- **当** 系统报告开机启动需要用户在系统设置中批准
- **那么** 应用显示需要批准的明确状态
- **并且** 不把该状态显示为已经启用

### Requirement: 可选子系统初始化失败不得终止整个应用

应用 MUST 将非核心本地子系统的初始化失败限制在对应 capability 内。若主窗口、设置、导航和其他独立服务仍可安全工作，系统不得因单个可选存储或服务失败而调用 `fatalError`、`preconditionFailure` 或主动终止进程。

#### Scenario: 可选本地存储初始化失败

- **当** 一个可选 capability 的本地存储无法创建、打开或初始化
- **那么** 应用仍显示主窗口并允许使用其他独立功能
- **并且** 受影响 capability 显示不可用状态和安全恢复建议
- **并且** 应用不自动删除、覆盖或迁移失败的数据

### Requirement: 后台活动必须遵守明确生命周期

应用 MUST 区分页面可见期间的连续采样、用户启用的应用级监控和有限时长操作。连续采样必须在所属页面停用后停止；应用级监控必须服从用户开关；有限时长操作必须支持既定取消语义并忽略晚到的旧结果。

#### Scenario: 离开系统监控页面

- **当** 用户离开系统监控页面
- **那么** 系统停止指标与应用用量连续采样
- **并且** 晚到结果不再更新停用页面的状态

#### Scenario: 离开剪贴板页面

- **当** 用户已明确启用剪贴板记录并离开剪贴板页面
- **那么** 应用级剪贴板监控按设置继续运行
- **并且** 用户关闭记录后监控立即停止

#### Scenario: 取消有限时长任务

- **当** 用户取消支持取消的搜索或扫描任务
- **那么** 服务释放任务持有的资源与 security scope
- **并且** 旧任务的晚到结果不会覆盖新状态
