# Design：add-disk-analysis

## 设计目标

这次 change 先交付“真实但克制”的磁盘信息：应用启动后尽快读取启动卷容量元数据，在 Dashboard 和磁盘分析页展示已用空间、可用空间和总容量；同时在磁盘分析页提供大文件列表，帮助用户识别主要空间占用来源。设计坚持只读、本地、低权限和可降级，不把容量概览或大文件展示扩展成高风险清理工具。

## 能力范围

首个阶段仅覆盖以下能力：

- 识别当前用户所在的启动卷或应用主工作卷。
- 读取卷级容量元数据，生成 `used / available / total` 摘要。
- 在 Dashboard 状态卡中于应用启动后显示容量摘要。
- 在磁盘分析页显示相同摘要，并提供显式手动刷新入口。
- 在磁盘分析页展示一组按大小降序排序的大文件结果，包含文件名、路径和大小等只读元信息。
- 当容量信息不可用、暂时失败或系统未返回完整数据时，展示降级文案。

以下能力明确留到后续 change：

- 目录树分析。
- 分类占用（缓存、日志、下载、应用残留等）。
- 清理建议与删除确认。
- 历史快照、趋势、后台定时刷新。
- 读取文件内容、哈希比对或重复文件聚类。

## 数据模型

建议新增共享值模型：

- `DiskCapacitySnapshot`
  - `volumeName`
  - `volumeIdentifier`（稳定但不暴露敏感路径）
  - `usedBytes`
  - `availableBytes`
  - `totalBytes`
  - `capturedAt`

- `DiskCapacityAvailability`
  - `.idle`
  - `.loading`
  - `.available(DiskCapacitySnapshot)`
  - `.unavailable(reason: DiskCapacityUnavailableReason)`

- `DiskCapacityUnavailableReason`
  - `.metadataNotReady`
  - `.resourceUnavailable`
  - `.unsupportedVolume`
  - `.unknown`

Dashboard 与磁盘分析页都只消费上述只读状态，不直接访问文件系统 API。

建议新增大文件模型：

- `LargeFileRecord`
  - `id`
  - `name`
  - `displayPath`（面向 UI 展示的路径字符串）
  - `sizeBytes`
  - `lastModifiedAt`（可选）
  - `sourceVolumeIdentifier`

- `LargeFileAvailability`
  - `.idle`
  - `.loading`
  - `.available([LargeFileRecord])`
  - `.unavailable(reason: LargeFileUnavailableReason)`

- `LargeFileUnavailableReason`
  - `.scanNotStarted`
  - `.resourceUnavailable`
  - `.permissionLimited`
  - `.unknown`

## 服务边界

新增 `DiskUsageService` 的首个最小职责：

- 提供读取当前目标卷容量摘要的异步接口。
- 返回值只包含卷级容量元数据与降级原因。
- 支持取消与显式刷新。
- 不缓存敏感路径，不记录用户目录名。
- 提供只读的大文件枚举接口，返回文件名、路径和大小等元信息。

建议接口方向：

- `loadStartupVolumeCapacity() async -> DiskCapacityAvailability`
- `refreshStartupVolumeCapacity() async -> DiskCapacityAvailability`
- `loadLargeFiles(limit: Int) async -> LargeFileAvailability`
- `refreshLargeFiles(limit: Int) async -> LargeFileAvailability`

服务实现应优先使用系统公开的卷资源元数据能力，例如卷总容量与可用容量字段推导已用空间；禁止通过递归目录统计来计算首页卡片数据。大文件展示允许在明确的用户可感知入口下进行本地文件枚举，但只读取文件元数据，不读取文件内容，不建立持久索引。

## 启动与刷新策略

### Dashboard

- 主窗口首次展示后开始异步读取容量摘要。
- 读取期间显示加载中占位，而不是“尚未扫描”。
- 成功后展示已用空间、可用空间和总容量。
- 失败时显示降级原因与“前往磁盘清理页重试”或等价提示。
- 加载态的三项容量位必须使用明确占位符，不展示旧的演示数值或静态百分比。
- 不可用态可以保留状态标签和安全说明，但不得显示任何会被理解为真实容量的数字。

### 磁盘分析页

- 页面出现时复用最近一次已知状态。
- 若当前仍是 `.idle`，则触发读取。
- 页面提供显式刷新动作，刷新更新卷级摘要和大文件列表。
- 不因进入页面自动启动深度目录分析或清理建议计算。
- 大文件列表可在页面内作为单独的只读模块出现，若列表未就绪则显示加载态或不可用原因。

### 状态一致性

- Dashboard 与磁盘分析页共享同一应用级只读状态源，避免同一时刻重复并发读取。
- 当手动刷新发生时，两处 UI 同步更新。
- 失败状态同样共享，避免首页和详情页文案不一致。
- 大文件列表状态只在磁盘分析 capability 内消费，但其刷新流程应与容量摘要协调，避免重复并发扫描。

## 首页展示更新

`main-dashboard` 的启动磁盘状态卡从“尚未扫描”调整为两态：

1. 可用：显示已用空间、可用空间、总容量，以及最近刷新时间或“刚刚更新”一类轻量说明。
2. 不可用：显示容量暂不可读取的原因与安全提示，不伪造数字。

首页卡片仍然是只读摘要卡，不承担扫描入口之外的敏感行为；Dashboard 下方快捷入口保持纯导航，不因状态卡刷新而附带扫描、清理或删除动作。

## 磁盘分析页展示

磁盘分析页在 Phase 0 先从纯占位升级为“只读概览页”：

- 标题区说明当前阶段仅展示启动卷容量概览。
- 主卡片展示已用空间、可用空间、总容量和简短说明。
- 大文件区块展示按大小降序的结果列表，至少包含文件名、路径和大小。
- 二级说明列出后续未实现能力，如目录分析、分类占用、清理建议。
- 若容量不可用，则以信息卡说明原因与保守降级策略。
- 若大文件列表不可用，则以区块级提示说明原因，不伪造文件结果。

## 日志与隐私

- 日志只记录成功/失败事件码与稳定错误类别。
- 不记录用户路径、卷挂载点、文件名、目录结构或容量明细对应的路径来源。
- 不因读取容量摘要而请求完全磁盘访问、辅助功能或其他额外权限。
- UI 可以展示用户文件路径，但日志与遥测不得记录这些路径或文件名。

## 风险与降级

### 卷元数据不可用

- 统一显示“容量信息暂不可用”及原因。
- 不回退为目录递归统计。
- 不把失败误报成 0 B 已用或 0 B 可用。

### 多卷与外置磁盘

- 本 change 仅要求展示启动卷或主工作卷摘要。
- 不在首页同时列出多个卷，避免信息复杂度和选择歧义。
- 若当前环境无法可靠判定目标卷，降级为不可用说明。

### 大文件枚举成本

- 大文件展示必须限制结果条数，例如仅展示 Top N。
- 应支持取消旧任务，避免用户频繁刷新时堆积枚举任务。
- 若枚举成本过高，应优先返回容量摘要，并让大文件区块独立进入加载或降级状态。

### 启动时延

- 容量读取必须异步执行，不阻塞主窗口首帧。
- 首页可先显示骨架或加载状态，随后更新为真实容量。

## 测试与验收方向

- 单元测试覆盖容量推导、大文件排序、降级状态和共享状态刷新。
- 视图测试覆盖 Dashboard 与磁盘分析页的可用/不可用/加载态，以及大文件区块展示。
- 人工验收覆盖：
  - 应用启动后首页展示真实已用/可用/总容量。
  - 磁盘分析页与首页数字一致。
  - 大文件列表按大小排序，并展示文件名、路径和大小。
  - 刷新后两处同步更新。
  - 无权限弹窗、无后台长时间扫描、无删除动作。
