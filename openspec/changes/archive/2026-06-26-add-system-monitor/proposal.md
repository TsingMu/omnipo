# Proposal：add-system-monitor

## 背景

Omnipo 主窗口“系统监控”入口目前仍是 Phase 0 占位页，用户进入后只能看到“Phase 0 暂未实现”文案，无法获得任何系统资源信息。Launcher 已提供“打开系统监控”命令，但点击后落到空页，与 macOS 自带“活动监视器”对比落差明显。

用户在排查卡顿、内存压力、网络占用或磁盘容量时希望快速看到关键指标，而不必切换到系统自带工具。Omnipo 作为本地优先、低开销的管家工具，适合提供一个简洁的、永远只读的系统资源概览页。

## 问题

- 系统监控入口落地为占位，缺乏最小可用的资源指标展示。
- 用户进入页面后没有可见反馈，无法判断是否在工作。
- 缺少与既有 `DiskUsageService` 等只读基础设施对齐的系统指标服务边界，未来扩展容易把监控页变成复杂控制台。
- 没有约束采样频率、生命周期与隐私边界，存在后台持续占用 CPU 或意外收集使用习惯的风险。

## 目标

- 新增 `system-monitor` capability 的 OpenSpec 文档，定义五维度只读资源概览：CPU、内存、能耗、磁盘、网络。
- 将主窗口“系统监控”页面从占位升级为多卡片只读概览页，每张卡片展示对应维度的当前值、简短说明与不可用降级文案。
- 提供统一的 `SystemMonitorService` 只读采样接口，支持异步批次、取消、可配置采样间隔，并在视图关闭或失活时停止后台采样。
- 明确五维度在 App Sandbox 内的可用边界：CPU/内存/磁盘容量/网络接口流量使用公开 API；能耗与实时磁盘 IOPS 无可靠公开 API 时降级说明，不伪造数据、不调用私有框架。
- 复用 `DiskUsageService` 与 `AppState` 已有的容量摘要，避免重复扫描与状态分裂。

## 非目标

- 不实现进程级监控（活动监视器的“进程”标签页、单进程 CPU/内存）。
- 不读取或解析其他应用的状态、文件、网络流量内容。
- 不上报任何指标到本地日志外的任何目标；指标仅用于当前 UI 展示。
- 不实现能耗瓦数精确测量：macOS 无公开整机能耗 API（SMC/PowerMetrics 需要特权或私有框架）。
- 不实现实时磁盘 IOPS、文件系统事件监听或后台常驻索引。
- 不修改任何系统设置、不请求额外隐私权限（辅助功能、输入监控、完全磁盘访问等）。
- 不在本 change 中实现历史趋势图表、阈值告警、导出报告或桌面通知。

## 受影响能力

- `system-monitor`：新增 capability，定义五维度只读概览、采样服务边界、降级策略与隐私约束。
- `application-foundation`：依赖容器装配新增 `SystemMonitorService`；主窗口“系统监控”入口替换占位页。

## 成功标准

- 主窗口“系统监控”页显示 CPU、内存、能耗、磁盘、网络五张概览卡片，且每张卡片在数据可用时展示真实当前值。
- 数据不可用的维度展示一致、可理解的降级原因（例如“能耗数据需系统特权”），不伪造数值。
- 采样由用户进入页面触发，离开页面或关闭主窗口时停止；不在后台持续运行。
- 采样频率可在设置中配置，默认值与可调范围在 design 中明确，且不可被设为 0 或负值。
- 所有采样、合并与展示完全在本地完成，不写入 OSLog 之外的任何持久化，日志不含具体数值细节之外的稳定事件码即可定位问题。
- 工程持续可编译，自动化测试覆盖采样器合并、降级状态映射、采样代次与取消；人工验收通过后才合并规范并归档 change。
